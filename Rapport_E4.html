Contexte & Objectifs

Périmètre du projet E4 Data Mobilité Paris

Présentation générale

Le projet E4 Data Mobilité Paris vise à construire un pipeline de données end-to-end autour de la mobilité douce parisienne. Il intègre des données open data, les orchestre via Apache Airflow 2.8.1, les stocke dans PostgreSQL 16 et les expose via une API FastAPI et un dashboard Streamlit, le tout déployé via Docker Compose.

Périmètre fonctionnel

  • Suivi en temps réel de la disponibilité des 1 510 stations Vélib' (rafraîchissement horaire)
  • Cartographie de 18 170 aménagements cyclables parisiens
  • Catalogue des événements culturels parisiens via le portail QFAP
  • Contrôle qualité automatisé quotidien — 4 contrôles + rapport consolidé
  • Statistiques consolidées quotidiennes — 3 domaines + rapport global
  • API REST sécurisée avec documentation Swagger auto-générée
  • Dashboard interactif avec cartes géolocalisées et graphiques

Stack technique

Apache Airflow 2.8.1 PostgreSQL 16 FastAPI Streamlit Docker Compose v2 Python 3.x pandas psycopg2

Étape Extract

Lecture des 4 fichiers CSV depuis le volume Docker /data/in/, encodage UTF-8, retour de DataFrames pandas avec gestion des erreurs de parsing.

import pandas as pd

def extract_csv(filepath: str) -> pd.DataFrame:
    df = pd.read_csv(filepath, encoding='utf-8', sep=';')
    return df

Étape Transform

  • Suppression des doublons (station_id comme clé naturelle)
  • Conversion des types : station_id → INTEGER, longueur → FLOAT, coordonnées → FLOAT
  • Nettoyage des valeurs nulles sur les champs obligatoires
  • Normalisation arrondissements : format court "11" au lieu de "75011"
  • Parsing des coordonnées géographiques (séparation lat/lon)
  • Conversion des timestamps actualisation_donnees

Étape Load

Connexion via psycopg2 avec les variables d'environnement Docker. Stratégie TRUNCATE + INSERT dans une transaction atomique — rollback automatique en cas d'erreur.

Incident — Verrou PostgreSQL (13/03/2026)

⚠️ Lors des premières exécutions parallèles, la tâche load_velib est restée bloquée en attente d'un verrou TRUNCATE détenu par transform_in_db. Résolu en identifiant les PIDs bloquants via pg_stat_activity et en appelant pg_terminate_backend(). Les DAGs ont ensuite été décalés (06h00, 06h30, 07h00, 08h00) pour éviter les conflits.
-- Diagnostic du verrou
SELECT pid, state, wait_event_type, left(query,80)
FROM pg_stat_activity
WHERE wait_event_type = 'Lock';

-- Résolution
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE wait_event_type = 'Lock';

Relation STATION (1) → DISPONIBILITE (N) : une station possède plusieurs snapshots horodatés de disponibilité (rafraîchissement horaire). La FK physique est volontairement absente pour optimiser les inserts haute fréquence. AMENAGEMENT_CYCLABLE et EVENEMENT_PARIS sont des entités autonomes.

Sécurisation minimale : blocage des requêtes destructrices

Pour garantir un usage lecture seule, l’application filtre les mots-clés SQL à risque (DROP, DELETE, TRUNCATE, INSERT, UPDATE, etc.). En cas de détection, l’exécution est refusée. 

def is_safe(q: str) -> bool:
    forbidden = ["drop", "delete", "truncate", "insert", "update", "alter", "create"]
    q_lower = q.lower()
    return not any(kw in q_lower for kw in forbidden)

Exemples de requêtes SQL (extraction + agrégation)

  • Top 20 stations Vélib les plus équipées (tri + limite) :
SELECT nom_station, nb_velos_disponibles,
       velos_electriques, velos_mecaniques, nb_bornettes_libres
FROM velib_disponibilite
ORDER BY nb_velos_disponibles DESC
LIMIT 20;
  • Événements par arrondissement (top 15) (agrégation COUNT(*)) :
SELECT code_postal,
       COUNT(*) AS nb_evenements
FROM evenements_qfap
WHERE code_postal IS NOT NULL
GROUP BY code_postal
ORDER BY nb_evenements DESC
LIMIT 15;
  • Types d’aménagements cyclables (agrégation + somme des longueurs) :
SELECT amenagement,
       COUNT(*)                        AS nb,
       ROUND(SUM(longueur)::numeric,2) AS km_total
FROM amenagements_cyclables
WHERE amenagement IS NOT NULL
GROUP BY amenagement
ORDER BY km_total DESC;

Ces requêtes sont ensuite visualisées automatiquement dans Streamlit (table + graphique), ce qui constitue une démonstration simple de l’usage “SQL → analyse” dans un contexte BI.
